# Mino Markdown Editor PRD 审核报告

> **审核日期**: 2026-05-28
> **审核对象**: `PRD - Mino Markdown Editor.md` (v0.1 DRAFT)
> **审核结论**: 产品定位清晰、技术方向正确，但存在 4 个必须修正的严重矛盾，需补齐基础功能后方可进入架构设计

---

## 🔴 严重问题（必须修正）

### 1. Pandoc Sidecar 与安装包体积目标直接矛盾

PRD 同时声称：

- 安装包 **< 15 MB**（1.3 核心价值 + 4.1 性能指标）
- F-14：使用 **Pandoc Sidecar** 导出 Word/PDF/HTML

**Pandoc 的 Windows 二进制约 50–100+ MB**。即使采用最小化构建，也不可能在包含 Pandoc 的同时将安装包压到 15 MB 以内。附录 A 的目录结构 `sidecars/pandoc/` 也证实了 Pandoc 会被打包进去。

**建议**：
- 方案 A：将 Pandoc 从安装包中剥离，改为用户首次使用导出功能时**按需下载**（参考 VS Code 的扩展安装模式）
- 方案 B：安装包体积指标改为**不含 Sidecar 的基础包 < 15 MB**，含 Pandoc 的完整包单独标注
- 方案 C：放弃 Pandoc，改用轻量级的前端库（如 markdown-it + html-docx-js）实现基础导出

### 2. F-01 验收标准与 4.1 性能指标自相矛盾

| 位置 | 指标 | 数值 |
|------|------|------|
| F-01 验收标准 | 10MB 文件加载 | **200ms** |
| 4.1 性能指标 | 10 万字文档加载 | **< 1 秒** |

10MB 的 Markdown 文件约 300–500 万字符，10 万字约 30 万字符（≈ 300KB–1MB UTF-8）。**两者差了 10–30 倍的体量，但更小的文件反而允许更长的加载时间**，这明显矛盾。

实际预估：10MB 的 Markdown 文件经过 Read → IPC → Remark 解析 → ProseMirror Schema 映射 → DOM 渲染的完整管线，**200ms 内完成几乎不可能**。

**建议**：
- F-01 验收标准改为：`1MB 文件在 500ms 内加载完成`（对应约 30 万字，与 4.1 对齐）
- 新增一条：`10MB 文件在 3 秒内加载完成`，作为压力测试基线

---

### 3. Rope 数据结构在 Rust 端对编辑性能几乎没有帮助

PRD 反复强调 Rope（ropey crate）支撑"微秒级大文件编辑"，但实际编辑流程是：

```
用户打字 → ProseMirror Transaction（JS 端）→ DOM 更新（WebView2 端）
```

**编辑发生在前端 ProseMirror 的 JS 内存中**，Rust 端的 Rope 只在文件读取和保存时参与。也就是说，Rope 解决的是 I/O 性能，而非编辑卡顿——而真正的性能瓶颈恰恰在前端 DOM 渲染。

**建议**：
- 明确区分两个性能问题：**文件 I/O 性能**（Rust Rope 解决）和 **编辑渲染性能**（前端视口消隐 + pos 闭包解决）
- 删除或修改"微秒级大文件编辑"的表述，因为该表述暗示 Rust Rope 能解决打字卡顿，这是误导

---

### 4. `display: none` 消隐方案与 R-06 风险缓解自相矛盾

| 位置 | 内容 |
|------|------|
| 5.2 优化策略表 | 将 `display: none` 装饰器列为正式策略 |
| R-06 风险缓解 | "仅用 `content-visibility`，不叠加 `display: none`" |

如果 R-06 的缓解措施是不用 `display: none`，那 5.2 就不该把它列为策略。

**建议**：5.2 表格中删除 `display: none` 方案，只保留 `content-visibility: auto` 作为唯一视口消隐策略。`display: none` 作为未来探索方向在风险登记中保留即可。

## 🟡 重要问题（建议修正）

### 5. Milkdown 与 Svelte 5 Runes 的集成是未经验证的技术路线

Milkdown 官方示例主要基于 React 和 Vue 的封装（`@milkdown/react`、`@milkdown/vue`），**没有官方的 Svelte 适配器**。Svelte 5 的 Runes 响应式系统与 ProseMirror 的命令式 Transaction 模型存在范式冲突：

- Svelte 5 Runes 是细粒度响应式（signal-based）
- ProseMirror 是命令式事务驱动（transaction-based）
- 两者如何桥接，目前**没有成熟的社区实践**

这比 R-05 "Svelte 5 生态不成熟"的风险更高——不是缺组件，而是核心集成路径不确定。

**建议**：在风险登记中新增一条 **R-07**，专门评估 Milkdown + Svelte 5 的集成风险，并考虑 ProseMirror 原生 API（不通过 Milkdown）作为降级方案。

---

### 6. 安装包 < 15 MB 的目标过于激进

拆解一下实际体积：

| 组成部分 | 预估体积 |
|---------|---------|
| Tauri v2 Rust 二进制 | ~2-4 MB |
| WebView2 前端资源（Svelte 5 + Milkdown + KaTeX + Prism.js + CSS + 主题） | ~5-8 MB |
| 图标、字体、本地化资源 | ~1-2 MB |
| **合计** | **~8-14 MB** |

这还是在**不含 Mermaid 渲染库**（~2-3 MB）的情况下。如果要支持 Mermaid，几乎必然超过 15 MB。

**建议**：将安装包目标调整为 **< 20 MB（基础包）**，并注明含 Mermaid 插件的完整包可能达到 25 MB。

---

### 7. MVP（P0）缺少关键编辑器功能

P0 需求列表中遗漏了以下任何编辑器都**必须有**的功能：

| 缺失功能 | 严重性 |
|---------|--------|
| **撤销/重做（Undo/Redo）** | 🔴 致命缺失 — 没有 Undo 的编辑器不可用 |
| **文件拖拽打开** | 🟡 用户期望——双击打开是基础操作 |
| **Windows 文件关联**（.md 默认打开） | 🟡 安装后应自动关联 |
| **最近打开文件列表** | 🟡 基础便利功能 |
| **查找/替换** | 🟡 被放到了 P2（F-21），但这对编辑器来说是 P0 |

特别是 **Undo/Redo**——ProseMirror 内置了强大的 Undo 管理器，但 PRD 只字未提，这是不可接受的。

**建议**：将 Undo/Redo、查找替换提升到 P0，文件拖拽打开和最近文件列表提升到 P1。

---

### 8. 安全性部分过于简略

4.3 安全性只列了 4 条，但遗漏了以下真实威胁场景：

| 威胁 | 说明 |
|------|------|
| Markdown 内嵌 HTML | 用户打开包含 `<script>` 或 `<iframe>` 的 .md 文件 |
| Mermaid 注入 | Mermaid 渲染可能执行恶意 SVG/JS |
| KaTeX 注入 | 复杂 LaTeX 公式可能触发 SSRF 或崩溃 |
| asset:// 路径穿越 | `$DOCUMENT/**` 范围过宽，可能暴露敏感文件 |

特别是 `assetScope: ["$DOCUMENT/**", "$DESKTOP/**"]` 这个配置**允许访问用户文档和桌面上的所有文件**，这与"文件访问沙箱"的主张矛盾。

**建议**：
- assetScope 限定为**当前打开文件所在目录及其子目录**
- 新增 Markdown 内嵌 HTML 的安全过滤策略
- 明确 Mermaid/KaTeX 的渲染安全边界

---

### 9. Phase 0 已勾选完成，但项目代码目录不存在

Phase 0 的 4 项全部标记为 `[x]`，但工作区中只有 `docs/` 和 `README.md`，**没有 `src-tauri/`、`src/`、`package.json` 等任何代码文件**。

**建议**：如果 Phase 0 尚未实际开始，将勾选改为 `[ ]`；如果是在其他仓库进行，需注明仓库地址。

| 维度 | 评分 | 说明 |

| 产品定位 | ★★★★★ | 清晰、差异化明确 |

## 核心建议

这份 PRD 的产品定位和技术方向是好的，但在落地层面需要解决 **4 个必须修正的严重矛盾**（Pandoc 体积、性能指标不一致、Rope 误导、display:none 自相矛盾），并补齐 **Undo/Redo 等缺失的基础功能**，才能进入下一阶段的架构设计。
