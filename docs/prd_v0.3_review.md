# PRD v0.3 审核 — Mino Markdown Editor

> 审核时间：2026-05-28

v0.3 相比 v0.1 有了大幅改进——之前的 15 个审核问题全部得到了回应。但新版本引入了新的内容和变更，也带来了一些新的问题。以下是 v0.3 的审核结果：

---

## 🔴 严重问题（必须修正）

### 1. `$CURRENT_FILE_DIR` 不是合法的 Tauri assetScope 变量

5.3 节的 `tauri.conf.json` 配置中使用了：

```json
"assetScope": ["$CURRENT_FILE_DIR/**"]
```

**Tauri 没有这个变量**。Tauri 内置的 scope 变量只有 `$HOME`、`$DOCUMENT`、`$DESKTOP`、`$DOWNLOAD`、`$APPDATA`、`$APPLOCALDATA`、`$APPCONFIG` 等。无法在静态配置文件中声明"当前文件所在目录"。

**解决方案**：assetScope 必须在运行时动态更新。当用户打开一个文件时，Rust 后端通过 Tauri 的 `fs_scope` API 动态授权该文件所在目录：

```rust
use tauri::scope::FsScope;

// 打开文件时动态授权
app.fs_scope().allow_directory(file_dir, true)?;
```

PRD 需要补充这一运行时授权机制的说明，否则安全模型无法落地。

---

### 2. 双栏模式 `replaceAll` 同步方案有严重缺陷

5.5.2 节描述同步机制为：

> "防抖 300ms 后调用 Milkdown 的 `editor.action(replaceAll(newMarkdown))` 触发预览重渲染"

`replaceAll` 意味着**每次同步都是全量替换整个文档**，存在三个问题：

| 问题 | 影响 |
|------|------|
| **光标/滚动丢失** | 预览面板每次全量替换后，滚动位置和用户正在查看的位置会重置 |
| **视觉闪烁** | 300ms 防抖后整篇文档重渲染，长文档会出现明显白屏闪烁 |
| **O(n) 复杂度** | 即使只改了一个字符，也要重新解析和渲染整篇文档 |

**建议**：改为增量同步方案——

1. CodeMirror 的变更通过 `ChangeSet` 获取差异（delta）
2. 将 delta 转换为 ProseMirror 的 `Step` 序列
3. 通过 `editor.view.dispatch(steps)` 增量更新 ProseMirror 状态

这样预览面板只更新变化的部分，不会丢失滚动位置。

---

### 3. P0 的查找/替换（F-12）与 CodeMirror 依赖存在冲突

P0 需求 F-12 要求"当前文件内文本搜索与替换，支持正则表达式"。

但附录 B 中 `@codemirror/search` 被列为 P1 依赖（双栏模式才引入 CodeMirror）。**P0 阶段只有 Milkdown/ProseMirror**，搜索功能必须在 ProseMirror 层实现——而 ProseMirror **没有内置搜索 API**。

**建议**：
- 明确 P0 的查找/替换使用 `prosemirror-search` 社区插件或自研 ProseMirror 搜索插件
- 将 `@codemirror/search` 从附录 B 的 P0 依赖中移除（或标注为"P1 双栏模式专用"）
- 补充说明 P0 阶段搜索的实现路径

---

### 4. 模式切换会丢失 Undo 历史，但文档未提及

5.5.4 描述模式切换流程：

> 即时预览 → 双栏：序列化 ProseMirror 文档为 Markdown 文本，填充 CodeMirror
> 双栏 → 即时预览：将 CodeMirror 文本重新注入 Milkdown

**每次切换都会导致 Undo 历史被清空**，因为 ProseMirror 和 CodeMirror 各自维护独立的 Undo 栈。用户切换模式后按 Ctrl+Z，会发现自己的操作历史全部消失——这是一个严重体验问题。

**建议**：
- 在文档中**明确声明这一限制**，并在 UX 上给出提示（如切换前 toast 提示"切换模式将清除撤销历史"）
- 长期方案：考虑将两套 Undo 栈序列化后互转，或在切换时将当前文档状态作为"快照"保存，确保至少能回退到切换前的状态

## 🟡 重要问题（建议修正）

### 5. 冷启动 < 500ms 的测量方式对用户有误导

4.1 节注明：

> "不含首次 WebView2 运行时初始化"

但从用户视角，**双击图标到编辑器可用**的时间才是真实体感。WebView2 首次冷启动初始化通常需要 200-400ms，加上 Rust 启动 + 前端渲染，实际体感冷启动可能在 600-900ms。

**建议**：提供两组指标：
- **技术冷启动**：< 500ms（不含 WebView2 初始化，用于 CI 基准测试）
- **用户体感冷启动**：< 1s（含 WebView2 初始化，完整链路）

---

### 6. Pandoc 从 GitHub Release 下载，中国用户可能无法访问

5.4 节描述 Pandoc 按需下载流程为"从 GitHub Release 下载"。中国大陆访问 GitHub 的稳定性差，下载大文件（60-100MB）经常超时或中断。

**建议**：
- 提供国内镜像源（如 Gitee Release、阿里云 OSS 等）
- 支持断点续传
- 允许用户手动指定本地 Pandoc 路径

---

### 7. 文件夹级全文搜索功能消失了

v0.1 中 F-21（P2）包含"当前文件/文件夹内正则搜索与替换"。v0.3 的 F-12（P0）只覆盖了"当前文件内"搜索，**文件夹级搜索**没有任何功能编号对应——它被悄悄删除了。

对于一个有"文件树侧栏"（F-18）的编辑器来说，文件夹级搜索是自然延伸。

**建议**：在 P2 中新增 **F-30 文件夹全文搜索**，或恢复原 F-21 的文件夹搜索部分。

---

### 8. Phase 1 工期从 3 周增至 4 周，但新增了两个 P0 功能

Phase 1 从 3 周延长到 4 周（+33%），但 P0 新增了 Undo/Redo（F-11）和查找/替换（F-12）。查找/替换在 ProseMirror 层需要自研或集成社区插件，工作量不小。4 周完成全部 P0 项偏乐观。

**建议**：Phase 1 调整为 5 周，或在 Phase 1 中将查找/替换降级为基础版（仅纯文本搜索，正则放到 P1）。
